1. 项目解决的核心痛点
企业知识难以“按任务”落地

传统知识库通常是关键词检索或简单问答，面对“需要按流程输出、要列步骤/检查清单、要满足边界条件”的场景，回答往往不够结构化、不可操作。

回答缺乏依据与可追溯性

企业文档篇幅大、版本多；如果回答不提供引用来源，业务人员无法判断结论来自哪些制度/条款/说明，从而难以合规采纳与复核。

检索与推理割裂，导致信息利用不足

单纯依赖模型生成会出现编造或偏离制度；单纯检索又难以形成完整“任务解决方案”。需要“检索证据 → 长链推理 → 任务化输出”的闭环。

面对材料缺失时无法正确升级/补齐

真实业务中材料往往不全。系统需要能判断“无法从当前材料给出确定结论”，并明确列出缺失信息清单与补充方向。

2. 核心逻辑流（检索 + 长链任务推理）
文档加载（Ingestion）

从 data/ 目录读取企业文档（txt / md / pdf），抽取文本并附带来源元数据（文件名、页码等）。

文本切分（Chunking）

将长文档按规则切成可检索的片段（chunk），同时保留来源信息，降低检索召回偏差与提示词长度压力。

向量化并构建索引（Indexing）

对每个 chunk 生成向量表示，写入 FAISS 向量库。这样后续可通过语义相似度快速定位相关证据段落。

用户提问触发检索（Retrieval）

用户输入问题后，系统用问题向量在 FAISS 中检索 top-K 片段，形成“检索材料集合”。

任务化长链推理（Task Reasoning）

将检索材料与用户问题共同拼装进提示词：

先判断任务结论/目标是否可从材料中确定
再给出任务步骤（1/2/3 或按业务流程拆解）
在关键结论处加入材料编号引用（如【材料2】），保证可追溯
补充注意事项/边界条件
若材料不足，输出缺失信息清单（需要哪些文档/字段/范围）
结果返回（Answer + Sources）

系统返回最终答案以及本次检索使用的材料来源列表，便于复核与审计。
